## 前言

这份文档不是技术规范、不是 API 手册、也不是答辩 PPT。它是我在做完毕业设计这个项目后，把自己脑子里的东西梳理出来给「将来某天回看代码的自己」看的一份地图。

我之前在做 HCM 实施助手时养成了写「为什么」而不是「是什么」的习惯——代码是结果，思路是原因，前者会过时，后者不会。这次毕业设计虽然题目相对简单，但里面藏着不少值得记录的小决策：为什么选 Django 而不是 FastAPI、为什么用 SQLite 而不是 MySQL、为什么坚持用 TF-IDF 不用更花哨的 BERT、为什么同一个推荐函数在求职者端和 HR 端要写两遍——这些「为什么」答辩时未必有时间细讲，但它们才是项目真正的肌理。

所以这份文档的特点是：每讲一段代码，我会同时讲清楚它解决什么业务问题、为什么选这种实现方式、有什么替代方案、踩过什么坑。

> 💡 毕业设计的价值不在于代码本身有多复杂，而在于你能不能讲清楚每一行代码为什么存在。

如果你是答辩老师偶然翻到这份文档，希望它能帮你快速理解我做了什么、思考了什么。如果你是同样在做求职招聘类毕设的同学，希望它能让你少走一点弯路。

## 第一章 项目总览


### 1.1 这是个什么项目

「基于内容推荐的求职与招聘系统」是一个面向高校毕业生与中小企业 HR 的轻量级招聘平台，核心特色是引入了基于内容的推荐算法，能根据求职者简历内容自动匹配最相关的岗位。

整个项目围绕三类角色的核心业务场景设计：

- **求职者端**——注册登录、简历维护、岗位浏览、智能推荐、在线投递、投递跟踪
- **企业 HR 端**——发布岗位、管理岗位、查看候选人（带匹配度）、更新候选人状态
- **管理员端**——平台数据统计、用户封禁/解封、岗位审核


### 1.2 项目数据

做完后客观盘点一下，这个项目的体量：

| 指标 | 数据 |
|---|---|
| 后端 Python 文件 | 约 8 个核心文件（含 4 个迁移） |
| Vue 前端页面 | 18 个（含 5 个求职者页 + 4 个 HR 页 + 4 个管理员页 + 公共页） |
| API 接口数量 | 22 个 |
| 数据库表 | 9 张（含审计、标签、通知三张扩展表） |
| 推荐算法核心 | TF-IDF + 余弦相似度，约 70 行 |
| 后端总代码量 | 约 800 行 Python |
| 开发周期 | 约 5 周（毕设答辩前） |

#### 为什么选 Django 不选 FastAPI

我做上一个项目时选了 FastAPI，这次反过来选了 Django，原因不是「赶时髦」而是项目性质不同。

HCM 实施助手是**纯算法服务**——核心是数据处理与文件 I/O，不涉及多角色、不涉及登录权限、不涉及结构化的数据持久化。FastAPI 的轻量恰好契合。

但这个毕设项目是**典型的 Web 应用**——多表关联、多角色权限、CRUD 占绝大多数业务、需要 Admin 后台辅助开发。Django 的几个关键优势恰好都是我需要的：

- **ORM 自带**——9 张表 + 外键关联，用 Django ORM 写起来一气呵成，不用单独引入 SQLAlchemy
- **Migration 自带**——表结构演化通过 `makemigrations` + `migrate` 自动管理，毕设答辩时改字段也不会乱
- **Admin 后台自带**——开发阶段塞数据、看数据全靠它，开发效率至少提升 30%
- **DRF 生态成熟**——`ModelSerializer` 几行代码就能给一个模型自动生成增删改查接口

> 💡 框架没有好坏，只有匹配与否。算法服务用 FastAPI，业务系统用 Django，这是我做完两个项目后最直观的体感。


#### 为什么用 SQLite 不用 MySQL

很多人觉得「毕设也得用 MySQL 才显得专业」。我反复权衡之后还是选了 SQLite，理由有三：

第一，**数据规模匹配**。毕设阶段的数据量最多几十个用户、几百个岗位、几千条投递记录，SQLite 在这个规模下的性能比 MySQL 还快（没有网络开销）。

第二，**部署成本低**。SQLite 是单文件数据库，整个项目 clone 下来就能跑，不需要装 MySQL 服务、不需要建库、不需要配置连接参数。答辩老师如果要本地复现，这个优势特别明显。

第三，**Django 透明切换**。Django ORM 屏蔽了数据库差异，未来如果要切换到 MySQL，改一行 `settings.py` 的 `ENGINE` 配置就行，业务代码完全不动。

代价：SQLite 在高并发写入时性能差。但毕设场景下根本到不了那个数量级。

## 第二章 架构设计


### 2.1 整体架构：前后端分离 + 单页应用

整个项目是标准的前后端分离 + SPA 架构。后端只管数据和接口，前端只管展示，两者靠 HTTP API 通信。

```
  ┌─────────────────────────────────┐
  │   前端（浏览器）                  │
  │   Vue 3 + Vite + Element Plus    │
  │   localhost:5173                │
  └────────────┬────────────────────┘
               │
               │  HTTP/JSON (Axios)
               │
  ┌────────────▼────────────────────┐
  │   后端（Django + DRF）            │
  │   Python + jieba + scikit-learn  │
  │   localhost:8000                │
  └────────────┬────────────────────┘
               │
               │  Django ORM
               │
  ┌────────────▼────────────────────┐
  │   SQLite 数据库（单文件）         │
  │   db.sqlite3                    │
  └─────────────────────────────────┘
```

这种架构的好处和上一个项目一致：

- 后端可以独立部署，未来想给小程序或 App 用也方便
- 前端可以独立迭代 UI，不影响业务逻辑
- 前后端可以分别用各自的最佳工具链


### 2.2 后端目录结构

```
recruitment_backend/
├── recruitment_backend/        # Django 项目级配置
│   ├── settings.py             # 项目配置（CORS、DRF、数据库等）
│   ├── urls.py                 # 顶级路由
│   ├── asgi.py / wsgi.py
├── recruitment/                # 应用级目录（业务全部在这里）
│   ├── models.py               # 数据模型（9 张表）
│   ├── serializers.py          # DRF 序列化器
│   ├── views.py                # 22 个 API 视图函数
│   ├── urls.py                 # 应用级路由
│   ├── admin.py                # Admin 后台注册
│   ├── recommender.py          # 推荐算法核心
│   ├── migrations/             # 数据库迁移文件（4 份）
│   └── apps.py
├── db.sqlite3                  # 数据库文件
└── manage.py                   # Django 入口
```


#### 为什么不分 services / utils / api 三层

参考我上一个项目 HCM 助手的目录结构，我刻意做了 api / modules / utils 三层。但这次没有。

原因：Django + DRF 的「函数视图（function-based view）」本身就是「薄薄一层」——一个 view 函数 = 一个接口，里面调用 Model 查询、调用 Serializer 序列化、返回 Response。逻辑链短，没必要再拆。

唯一「需要分层」的代码是推荐算法——它本身是一个相对独立的算法模块，依赖 sklearn 和 jieba，被两个视图调用（求职者端的 `recommend_jobs` 和 HR 端的 `job_candidates` 计算 match_score 时）。所以我把它单独拆到 `recommender.py`。

> 💡 分层不是越多越好。每多一层，就多一份「这段代码在哪个文件」的认知负担。判断标准：被复用 2 次以上 → 单独拆；只被用 1 次 → 留在 view 里。

#### 嵌套路由的妙用：三套角色，三个 Layout

整个前端只有三个「外壳」组件：`SeekerHome.vue`、`HrHome.vue`、`AdminHome.vue`。每个外壳提供该角色专属的顶部导航/侧边菜单，里面用 `<router-view />` 装载具体页面：

```js
{
  path: '/seeker',
  component: SeekerHome,          // 外壳
  children: [
    { path: '',             component: SeekerDashboard },
    { path: 'jobs',         component: JobList },
    { path: 'resume',       component: MyResume },
    { path: 'recommend',    component: RecommendJobs },
    { path: 'applications', component: MyApplications },
    { path: 'jobs/:id',     component: JobDetail },
  ]
}
```

这种「外壳 + 嵌套页面」的模式有两个好处：

第一，**角色边界一目了然**。看 router 配置就能知道某个页面属于哪个角色，找代码秒级定位。

第二，**菜单与路由天然绑定**。每个外壳里的菜单项直接 `router.push('/seeker/jobs')`，不需要额外维护「菜单—路由映射表」。

## 第三章 数据库与模型设计


### 3.1 9 张表的全景

整个系统的数据模型由 9 张表构成，包括 6 张核心业务表 + 3 张扩展表：

| 表名 | 中文名 | 角色 | 字段数 |
|---|---|---|---|
| user | 用户表 | 核心 | 7 |
| company | 企业表 | 核心 | 7 |
| resume | 简历表 | 核心 | 19 |
| job | 岗位表 | 核心 | 12 |
| application | 投递记录表 | 核心 | 6 |
| system_config | 系统配置表 | 核心 | 3 |
| audit_log | 岗位审核日志表 | 扩展 | 5 |
| job_tag | 岗位标签表 | 扩展 | 3 |
| notification | 消息通知表 | 扩展 | 6 |

其中 6 张核心表覆盖了基础业务流转，3 张扩展表是按答辩老师建议补充的——「审计、标签、通知」是企业级 SaaS 系统的标配，加上它们能让数据库设计看起来更完整、更「像那么回事」。


### 3.2 关键模型：User 表的「三合一」设计

`User` 表通过一个 `role` 字段同时承载三类角色，没有按角色拆成三张表：

```python
class User(models.Model):
    ROLE_CHOICES = [
        ('seeker', '求职者'),
        ('hr', '企业HR'),
        ('admin', '管理员'),
    ]
    username = models.CharField(max_length=50, unique=True)
    password = models.CharField(max_length=255)
    role = models.CharField(max_length=10, choices=ROLE_CHOICES)
    phone = models.CharField(max_length=20, blank=True)
    email = models.CharField(max_length=100, blank=True)
    is_active = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
```


#### 为什么不按角色拆表

我做模型设计时一度想过 `Seeker` / `HrUser` / `AdminUser` 三张表。后来放弃了，原因有三：

第一，**三类角色共用「注册—登录—封禁」逻辑**。如果拆三张表，登录接口要先猜用户在哪张表里、封禁接口要按角色查不同表，工作量翻倍。

第二，**角色字段是分类，不是身份**。Stack Overflow 上有句话：「如果一个东西只是被分类，那么用枚举字段；如果它有独立属性，才拆表。」企业 HR 有独立属性（所属企业），但这部分属性放在 `Company` 表里通过外键关联即可，`User` 表本身不需要拆。

第三，**Django 后续可扩展**。如果将来求职者需要独立属性（如「会员等级」），可以用 Django 的 `OneToOneField` 扩展出一张 `SeekerProfile` 表，不破坏原有结构。

代价：每次查询都要带上 `role` 字段做过滤。但这个代价远小于拆表带来的复杂度。


### 3.3 关键模型：Resume 表的「多对一 + 单点启用」设计

简历表的设计有两个细节值得展开：

```python
class Resume(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name='resumes')
    title = models.CharField(max_length=100, default='默认简历')
    is_active = models.BooleanField(default=False)
    # ... 19 个业务字段
```


#### 细节一：一个求职者可以有多份简历

`user` 字段是 `ForeignKey` 不是 `OneToOneField`——一个求职者可以维护多份简历（如「Java 后端」和「Vue 前端」两份）。这符合真实求职习惯：投不同岗位用不同简历，是行业惯例。


#### 细节二：单点启用机制

任何时刻只允许一份简历的 `is_active = True`。「启用简历」是个状态约束，不是字段约束——靠业务代码维护。看 `activate_resume` 视图：

```python
@api_view(['POST'])
def activate_resume(request, resume_id):
    resume = Resume.objects.get(id=resume_id)
    # 先把该用户所有简历置为非启用
    Resume.objects.filter(user_id=resume.user_id).update(is_active=False)
    # 再启用当前这份
    resume.is_active = True
    resume.save()
    return Response({'msg': '启用成功'})
```

两行代码完成「单点启用」语义。但这里有个**并发陷阱**——如果两个请求同时启用不同简历，可能出现两份同时为 active 的状况。在毕设场景下这不会发生（单用户单浏览器），但生产环境需要用数据库事务或 `select_for_update` 锁。我在文档里明确标注了这一点，是为了展示「知道边界在哪」。


### 3.4 关键模型：Job 表的状态机

岗位的全生命周期通过 `status` 字段管理，定义了五种状态：

```python
STATUS_CHOICES = [
    ('pending',  '待审核'),   # HR 提交，等管理员审核
    ('approved', '已发布'),   # 管理员审核通过，求职者可见
    ('rejected', '已拒绝'),   # 管理员审核拒绝
    ('offline',  '已下线'),   # HR 主动下线 或 系统因招满自动下线
    ('closed',   '招聘结束'), # 招聘任务完成
]
```


#### 状态转换矩阵

| 当前状态 | 允许转换到 | 触发方 |
|---|---|---|
| pending | approved / rejected | 管理员审核 |
| approved | offline / closed | HR 主动操作或系统自动 |
| rejected | （终态，不可恢复）| — |
| offline | pending（HR 重新发布）| HR |
| closed | （终态）| — |

这个矩阵答辩时如果被问「状态怎么流转」可以直接画在白板上。每条边都对应一个真实业务场景。


### 3.5 Migration 演化：4 份迁移文件讲了什么故事

`migrations/` 目录有 4 份迁移文件，按时间顺序还原了整个模型的演化：

| 迁移 | 干了啥 | 当时为什么 |
|---|---|---|
| 0001_initial | 创建 user/job/resume/application/system_config 5 张表 | 项目初始化 |
| 0002 | 添加 company 表 + 重新组织 application 的关联 | 引入企业概念后发现 HR 必须挂在企业下 |
| 0003 | resume 增加 project_exp、certificate_exp 两个字段 | 答辩老师反馈：缺少项目经历就不像真简历 |
| 0004 | resume 增加 created_at / is_active / title 等 5 个字段 | 实现「多简历 + 单点启用」需求 |

> 💡 不要害怕看 migration 文件——它们是项目演化最忠实的记录。比 git log 还能讲故事。

## 第四章 角色与权限模型


### 4.1 三类角色的能力边界

```
┌──────────────────────────────────────────────────────────┐
│                       系统全部能力                          │
├────────────────┬────────────────┬────────────────────────┤
│   求职者 seeker  │   企业 HR hr    │    管理员 admin         │
├────────────────┼────────────────┼────────────────────────┤

```

每类角色看不到也用不上其他角色的功能。前端通过路由分组隔离 (`/seeker/*`、`/hr/*`、`/admin/*`)，后端通过视图层判断 `role` 字段隔离。


### 4.2 一个被简化的设计：没有用 JWT

按生产标准应该用 JWT 或 Session 做身份认证。但这个毕设里我用了一个简化方案：

- 登录成功后，前端把 `user_id` / `username` / `role` 三个字段塞进 `localStorage`
- 每次请求带 `user_id` 作为参数（query 或 body）
- 后端用 `user_id` 查数据

```js
// 前端登录成功后
localStorage.setItem('user_id', res.data.user_id)
localStorage.setItem('username', res.data.username)
localStorage.setItem('role', res.data.role)
```


#### 这个方案有什么问题

明显的问题：**不安全**。前端 `localStorage` 可以被篡改，恶意用户改一下 `user_id` 就能假装成别人。也没做 CSRF 防护。

为什么还是这么做：**毕设场景下不需要**。答辩验证的是业务功能能不能跑通、推荐算法对不对，而不是「能不能扛得住攻击」。把精力花在 JWT 上反而会让推荐算法部分变薄。

我在文档里明确写明：「**生产环境必须升级为 JWT + HTTPS + CSRF Token 三件套**」。这是「知道但故意没做」，不是「不知道」。

> 💡 毕设最危险的不是「不会做的事」，而是「啥都想做但啥都做不深」。每个简化都要有明确理由。


### 4.3 密码加密：MD5 够用吗

注册时密码用 MD5 加密存储：

```python
def create(self, validated_data):
    validated_data['password'] = hashlib.md5(
        validated_data['password'].encode()
    ).hexdigest()
    return super().create(validated_data)
```


#### MD5 已经不安全了，为什么还在用

MD5 在密码学上确实被证明不安全（容易被彩虹表反查）。生产环境标准做法是 `bcrypt` 或 `argon2`，它们带盐（salt）且计算慢，专门为抗破解设计。

为什么没换：和上面一样，毕设阶段简化处理。但答辩时如果被问「为什么不用 bcrypt」，我会这样回答：

- **认识到**：MD5 在密码学上已被证明不安全
- **替代方案**：bcrypt（带盐 + 计算慢）或 argon2（更新一代）
- **当前选择 MD5 的理由**：毕设答辩重点验证业务逻辑，且数据库里只有几条种子数据，不存在被反查的实际风险
- **生产环境升级路径**：在 `serializers.py` 把 `hashlib.md5(...)` 改为 `bcrypt.hashpw(...)` 即可，业务代码不动

这种「知道但简化」的态度比「不知道」要好得多。

## 第五章 简历与岗位模块详解


### 5.1 业务背景：简历的 19 个字段

简历表有 19 个业务字段，按用途分四类：

**身份与基本信息**：real_name、school、major、degree

**经历类（文本字段，推荐算法的主要原料）**：education_exp、work_exp、project_exp、certificate_exp、self_intro

**能力与意愿**：skills、expect_job、expect_city、expect_salary

**元数据**：title（简历名）、is_active、created_at、updated_at、content_text（推荐用拼接文本，预留字段）

为什么把字段切这么细：每个字段在简历填写表单里都是独立输入框，分得越细越方便求职者填写、也越方便推荐算法识别。


### 5.2 关键设计：简历文本的拼接策略

推荐算法的输入是「文本」，而简历有 19 个字段。怎么把它们拼成一段文本？看 `build_resume_text` 函数：

```python
def build_resume_text(resume):
    """把简历字段拼接成推荐文本"""
    parts = [
        resume.skills or "",
        resume.expect_job or "",
        resume.work_exp or "",
        resume.self_intro or "",
        resume.education_exp or "",
        getattr(resume, "project_exp", "") or "",
        getattr(resume, "certificate_exp", "") or "",
        resume.major or "",
        resume.degree or "",
        resume.expect_city or "",
    ]
    return tokenize(" ".join(parts))
```


#### 字段顺序有讲究

10 个字段按「重要性」排序，最重要的（skills、expect_job）排前面：

- `skills` 第一位：这是简历里信号最强的字段，直接对应岗位的 `skill_required`
- `expect_job` 第二位：求职者明确表达的求职意向，方向性强
- 然后是经历类字段（work_exp、self_intro、education_exp、project_exp、certificate_exp）
- 最后是元信息（major、degree、expect_city）

但事实上，**TF-IDF 算法不关心字段顺序**——它是「词袋模型」（Bag of Words），把所有词当成一个无序集合处理。所以这个排序更多是给「看代码的人」的提示，告诉他这些字段在业务上的重要性。


#### getattr 的双重容错

```python
getattr(resume, "project_exp", "") or ""
```

为什么不直接写 `resume.project_exp or ""`？

因为 `project_exp` 是迁移 0003 才加的字段。早期数据库里的简历对象**没有这个属性**，直接访问会抛 `AttributeError`。`getattr(obj, name, default)` 第三个参数是默认值，属性不存在时返回 `""` 而不是抛异常。

后面再叠一个 `or ""`：处理「属性存在但值是 None」的情况（数据库字段允许 null 时）。

两层容错是「**向前兼容**」和「**防御性编程**」的混合体——既兼容旧数据，又防范字段为 null 的情况。

> 💡 写库代码时养成 `getattr(obj, name, default) or default` 的习惯，能省掉无数次半夜的「AttributeError: 'Resume' object has no attribute ...」。


### 5.3 岗位发布的状态流转

HR 发布岗位时，岗位默认 `status='pending'`（待审核）：

```python
@api_view(['POST'])
def job_create(request):
    serializer = JobSerializer(data=request.data)
    if serializer.is_valid():
        serializer.save()  # 默认 status='pending'
        return Response({'msg': '职位发布成功，等待审核'})
```

然后管理员通过 `review_job` 接口将其改为 `approved` 或 `rejected`：

```python
@api_view(['POST'])
def review_job(request, job_id):
    job = Job.objects.get(id=job_id)
    job.status = request.data.get('status')  # 'approved' 或 'rejected'
    job.save()
    return Response({'msg': '审核完成'})
```

只有 `status='approved'` 的岗位才会出现在求职者的岗位列表和推荐结果中：

```python
jobs = Job.objects.filter(status='approved')
```

这个简单的状态过滤实现了完整的「**审核制岗位池**」语义。


#### 一个被砍掉的功能：审核理由

最初的版本里 `review_job` 接口还要接收 `reject_reason`（拒绝理由），保存到 `audit_log` 表里。后来在和指导老师讨论时砍掉了——理由：

- 增加这个字段会让审核界面多一个输入框、审核流程多一步
- 审计日志表本身已经存在，可以补到日志里而不是改 Job 表
- 毕设阶段优先保证主流程跑通

最终方案：`audit_log` 表的设计保留了 `reject_reason` 字段，但实际审核时只调用 `review_job` 不写日志。文档里说明「**预留扩展位**」。这种「设计上完整、实现上简化」是毕设阶段的常见取舍。

## 第六章 推荐算法模块详解


### 6.1 业务背景：为什么要做推荐

这是整个毕设的「特色点」也是题目的「内容」。

不做推荐的话，求职者只能看「岗位列表」，靠关键词搜索。这个交互在岗位数少时还行，岗位多了之后体验崩塌——求职者要翻无数页才能找到适合的。

引入推荐之后，求职者打开「推荐岗位」页就能看到按匹配度从高到低排序的岗位卡片，每张卡片标注一个百分比的匹配度。这个设计直接对标 BOSS 直聘、智联招聘等头部产品的核心交互。


### 6.2 算法选型：为什么是 TF-IDF + 余弦相似度

推荐算法主要有三大类：

| 类别 | 代表方法 | 适合场景 |
|---|---|---|
| 协同过滤 | UserCF、ItemCF | 有大量用户行为数据（点击/收藏/购买） |
| 基于内容 | TF-IDF + 余弦、Word2Vec、BERT | 物品有结构化文本属性 |
| 混合推荐 | LinkedIn 早期模型 | 数据规模够大，可融合多种信号 |


#### 我的取舍

毕设场景下用户行为数据极少（几乎没人真的点击/投递过几百次），协同过滤根本跑不起来。剩下基于内容的方法，又分两档：

- **轻量档**：TF-IDF + 余弦相似度 → 不需要训练、可解释性强、毕设场景秒级出结果
- **重量档**：BERT 编码 + 语义相似度 → 需要预训练模型、显存占用大、解释性弱

我选轻量档，理由有四点（详细论述见专门的《推荐算法设计与实现说明》文档）：

第一，**数据规模匹配**：几十个用户、几百个岗位，TF-IDF 完全够用。

第二，**冷启动友好**：新用户没行为数据时，TF-IDF 依然可以基于简历文本直接生成推荐。

第三，**可解释性强**：「岗位 3 和你的简历共同包含 Python、Django、MySQL，所以匹配度高」——这种解释直接来自向量的非零分量，答辩时一目了然。

第四，**部署成本低**：sklearn + jieba 加起来不到 100M，本地几行代码搞定，不需要 GPU。


### 6.3 核心代码：recommender.py 全文

整个推荐模块的核心代码不到 70 行，结构清晰：

```python
import jieba
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity
from .models import Job


def tokenize(text):
    """中文分词"""
    text = text or ""
    return " ".join(jieba.cut(text))


def build_resume_text(resume):
    """把简历字段拼接成推荐文本"""
    parts = [
        resume.skills or "",
        resume.expect_job or "",
        resume.work_exp or "",
        resume.self_intro or "",
        resume.education_exp or "",
        getattr(resume, "project_exp", "") or "",
        getattr(resume, "certificate_exp", "") or "",
        resume.major or "",
        resume.degree or "",
        resume.expect_city or "",
    ]
    return tokenize(" ".join(parts))


def build_job_text(job):
    """把职位字段拼接成推荐文本"""
    parts = [
        job.title or "",
        job.description or "",
        job.skill_required or "",
        job.city or "",
        job.degree_required or "",
        job.exp_required or "",
        job.job_type or "",
    ]
    return tokenize(" ".join(parts))


def get_recommendations(resume):
    """根据简历推荐职位"""
    if not resume:
        return []

    resume_text = build_resume_text(resume)
    jobs = list(Job.objects.filter(status='approved'))
    if not jobs:
        return []

    job_texts = [build_job_text(job) for job in jobs]

    corpus = [resume_text] + job_texts
    vectorizer = TfidfVectorizer()
    tfidf_matrix = vectorizer.fit_transform(corpus)

    resume_vec = tfidf_matrix[0]
    job_vecs = tfidf_matrix[1:]
    scores = cosine_similarity(resume_vec, job_vecs)[0]

    ranked = sorted(zip(jobs, scores), key=lambda x: x[1], reverse=True)
    return ranked
```


### 6.4 三个关键决策的逐一拆解


#### 决策一：为什么把简历和所有岗位拼成一个语料库

看这一行：

```python
corpus = [resume_text] + job_texts
vectorizer = TfidfVectorizer()
tfidf_matrix = vectorizer.fit_transform(corpus)
```

简历是 1 篇文档，岗位是 N 篇文档，**一起**传给 TfidfVectorizer。

如果分开传会怎样？比如对简历单独调一次 `fit_transform`、对岗位列表再调一次：

- 两次 fit 会产生**两个不同的词典**——简历的词典可能没有「Spring」（如果简历没写），岗位的词典可能没有「Vue」（如果某个岗位没写）
- 词典不同，向量维度也不同，**不能直接做余弦相似度**

把所有文档塞到同一个 corpus 里，保证它们在**同一个特征空间**中表达，向量才能直接比较。这是工程上最容易踩的坑。


#### 决策二：jieba 分词为什么用空格连接

```python
return " ".join(jieba.cut(text))
```

`jieba.cut` 返回一个生成器，里面是分好的词。为什么要 `" ".join` 用空格连成字符串？

因为 sklearn 的 `TfidfVectorizer` 默认按**空白字符**切分词。中文原文「Python Django Vue」对它来说是一个整体（中间没空格），它根本识别不出多个词。

正确流程是：
1. `jieba.cut("Python Django Vue后端开发")` → 生成器 `['Python', ' ', 'Django', ' ', 'Vue', '后端', '开发']`
2. `" ".join(...)` → 字符串 `"Python   Django   Vue 后端 开发"`
3. 传给 `TfidfVectorizer` → 按空格切，得到词列表 `['Python', 'Django', 'Vue', '后端', '开发']`

不用 jieba 分词的话，sklearn 看到的是一整团没有边界的中文字符，完全没法处理。


#### 决策三：list() 强制实例化的目的

```python
jobs = list(Job.objects.filter(status='approved'))
```

为什么要 `list(...)` 强制把 QuerySet 转成列表？

Django ORM 的 `QuerySet` 是**懒加载**的——你写 `Job.objects.filter(...)` 不会立即查数据库，要等到真正遍历或索引时才查。

如果不加 `list(...)`，后面的代码会发生：

1. `job_texts = [build_job_text(job) for job in jobs]` → 第一次访问 jobs，查数据库
2. `sorted(zip(jobs, scores), ...)` → 第二次访问 jobs，**再查一次数据库**

加上 `list(...)`，QuerySet 立刻实例化为 Python 列表，后续访问都是内存操作，**只查一次库**。

> 💡 ORM 的懒加载是个性能陷阱——看起来没问题的代码可能偷偷查了 10 次库。养成「需要复用的 QuerySet 立即转 list」的习惯。


### 6.5 一个被复制粘贴的函数：calc_resume_job_score

`views.py` 里有个看着和 `get_recommendations` 很像的函数：

```python
def calc_resume_job_score(resume, job):
    if not resume or not job:
        return 0
    # ... 拼接 resume_text 和 job_text ...
    corpus = [resume_text, job_text]
    vectorizer = TfidfVectorizer()
    tfidf_matrix = vectorizer.fit_transform(corpus)
    score = cosine_similarity(tfidf_matrix[0], tfidf_matrix[1])[0][0]
    return round(float(score) * 100, 1)
```

它在 HR 端的「查看候选人列表」里被调用，给每个候选人标注一个匹配度百分比。


#### 为什么不复用 get_recommendations

第一眼看上去这是「**重复代码**」——文本拼接逻辑和 TF-IDF 计算流程几乎一样。我也想过抽公共函数。但仔细分析后没动：

| 维度 | get_recommendations | calc_resume_job_score |
|---|---|---|
| 输入 | 一份简历 | 一份简历 + 一个岗位 |
| 语料库 | 1 简历 + N 岗位 | 1 简历 + 1 岗位 |
| 输出 | 排序后的列表 | 单个匹配度百分比 |
| 用途 | 求职者端推荐 | HR 端候选人匹配度展示 |

如果硬要复用，就要写一个「**多用途**」的函数，参数里塞一堆控制开关（是返回排序列表还是单个分？是百分比还是 0-1？是否过滤 approved 状态？）。这种「**配置爆炸**」的函数比两份独立函数还难维护。

最终决定：**保留两份**。但在每份函数顶部加注释说明它们的相似性和差异，避免后人误以为只是 bug。

> 💡 「Don't Repeat Yourself」是好原则，但它有个隐藏前提：**重复的是相同业务逻辑**。如果两段相似的代码服务于两个独立的业务场景，强行合并往往得不偿失。


### 6.6 推荐算法的局限性（必须坦白）

毕设答辩老师最爱问的问题之一就是：「**你这个算法的缺点是什么**」。提前想清楚比当场支支吾吾要好得多。

**第一个缺点**：TF-IDF 是词袋模型，**完全忽略词序**。简历里「不熟悉 Java」和「熟悉 Java」会被识别成同一个意思，因为词袋里只统计「不熟悉、Java、熟悉」三个词的出现次数。

**第二个缺点**：**没有语义理解**。如果简历写「机器学习」，岗位写「machine learning」，TF-IDF 认为这是完全不同的两个词。BERT 这类预训练模型能解决，但成本太高。

**第三个缺点**：**每次重算**。当前实现每次推荐请求都重新构建 TF-IDF 矩阵。岗位池一旦上万，性能就崩。生产环境应该把岗位向量缓存到 Redis，只对新增/修改的岗位增量计算。

**第四个缺点**：**没用上行为数据**。求职者历史投递过哪些岗位、收藏过哪些岗位，这些都是宝贵的偏好信号，当前完全没利用。要补这一块需要加协同过滤层，工作量翻倍。

把这四个缺点写在文档里、答辩时主动讲出来，比被老师追问要从容。「**知道边界**」永远比「**假装完美**」加分。

## 第七章 候选人匹配与状态流转


### 7.1 一个有趣的对称性

求职者端的「推荐岗位」和 HR 端的「候选人列表」是**对称的**两个视图：

- **求职者视角**：给定一份简历，从所有岗位中挑选最匹配的，按匹配度排序
- **HR 视角**：给定一个岗位，从所有投递了这个岗位的简历中，计算每份的匹配度

数学上是同一个公式，工程上是同一段算法。但 UI 呈现完全不同：

- 求职者看到的是岗位卡片，每张标「匹配度 74.3%」
- HR 看到的是候选人列表行，每行带「匹配度 74.3%」+ 候选人姓名 + 学校 + 期望岗位


### 7.2 候选人状态机

`Application` 表的 `status` 字段实现了候选人状态机：

```python
STATUS_CHOICES = [
    ('pending',   '待查看'),   # 投递成功，HR 还没看
    ('viewed',    '已查看'),   # HR 打开过简历
    ('interview', '面试'),      # 进入面试流程
    ('rejected',  '不合适'),    # 简历被淘汰
    ('hired',     '录用'),      # 已录用
]
```

完整的转换路径：

```
pending → viewed → interview → hired
              ↓        ↓
           rejected  rejected
```

任何状态都可以转换到 `rejected`，因为「淘汰」可以发生在任意阶段。`hired` 是终态。


### 7.3 数据丰富的 Serializer

`ApplicationSerializer` 是整个项目里**最复杂的序列化器**，它通过反向关联把多个表的字段聚合到投递记录里：

```python
class ApplicationSerializer(serializers.ModelSerializer):
    # 从 job 表取
    job_title = serializers.CharField(source='job.title', read_only=True)
    city = serializers.CharField(source='job.city', read_only=True)
    salary_min = serializers.IntegerField(source='job.salary_min', read_only=True)
    salary_max = serializers.IntegerField(source='job.salary_max', read_only=True)
    
    # 跨两层关联从 company 表取
    company_name = serializers.SerializerMethodField()
    
    # 从 user / resume 表取候选人信息
    candidate_name = serializers.SerializerMethodField()
    candidate_username = serializers.CharField(source='user.username', read_only=True)
    school = serializers.CharField(source='resume.school', read_only=True)
    major = serializers.CharField(source='resume.major', read_only=True)
    degree = serializers.CharField(source='resume.degree', read_only=True)
    expect_job = serializers.CharField(source='resume.expect_job', read_only=True)
    resume_title = serializers.CharField(source='resume.title', read_only=True)
```

为什么写这么多 `source='xx.yy'`？因为前端每次渲染一行投递记录，需要显示**岗位标题、企业名、城市、薪资、候选人姓名、学校、专业、学历**等十几个字段——它们分散在 4 张表里（job、company、user、resume）。

如果前端每条记录都单独发一次「查岗位详情」「查企业详情」请求，就是典型的 **N+1 查询问题**。后端这样在序列化器层一次聚合，前端拿到一个完整对象就能渲染整行。

配合视图层的 `select_related`：

```python
apps = Application.objects.filter(job_id=job_id).select_related(
    'user', 'resume', 'job', 'job__company'
)
```

把跨表关联在一次 SQL JOIN 里搞定，避免循环里查库。

> 💡 DRF 的 `source` + Django ORM 的 `select_related` 是「**避免 N+1 查询**」的黄金组合。学会这一对就能写出性能合格的列表接口。


### 7.4 一个隐蔽的边界条件：删除已启用简历

`delete_resume` 视图里有这样一段：

```python
user_id = resume.user_id
was_active = resume.is_active
resume.delete()

if was_active:
    next_resume = Resume.objects.filter(user_id=user_id).order_by(
        '-updated_at', '-created_at'
    ).first()
    if next_resume:
        next_resume.is_active = True
        next_resume.save()
```

业务规则：求职者删除当前启用的简历后，系统自动启用其**最近更新**的另一份简历（如果有的话）。这避免了出现「该用户有简历但没有启用简历」的状态。


#### 为什么用 -updated_at 而不是 -created_at

```python
.order_by('-updated_at', '-created_at')
```

主排序键是 `updated_at`——最近编辑过的简历优先启用，因为这通常是求职者最在意的那一份。如果两份简历更新时间一样，再按创建时间倒序。

这是个**用户体验**层面的小细节。要不要做？做。会不会有人注意到？大概率不会。但它和「不做」之间的差别就是「这个产品看起来用心了」vs「这个产品看起来没考虑过这个场景」。

## 第八章 前端工程实践


### 8.1 Element Plus 的取舍

整个前端的 UI 几乎全部基于 Element Plus 组件——表单（`el-form`）、表格（`el-table`）、按钮（`el-button`）、消息提示（`ElMessage`）、对话框（`el-dialog`）、菜单（`el-menu`）……几乎不写原生 HTML 组件。

为什么这么依赖 Element Plus：

- **不需要从零写组件库**：作为一个人项目，把时间花在 CSS 调样式上是浪费
- **企业级风格统一**：Element Plus 的视觉风格对标阿里飞冰，看起来「正经」
- **国内文档好**：Element Plus 的中文文档质量比 Ant Design Vue 还高


#### 但也踩了几个坑

第一，**字符串图标 prefix-icon 不生效**：
```html
<el-input prefix-icon="User" />   <!-- 不行 -->
```
Element Plus 2.x 要求图标用组件方式引入，但我前期写代码时没注意，导致登录页前缀图标显示为「User」三个字。后来发现要么按文档全量注册图标组件，要么干脆不要图标，最终选了后者——少一个图标不影响功能。

第二，**Dialog 的双向绑定**：`<el-dialog v-model="visible">` 而不是 `visible="true"`。第一次写时少了 `v-model`，按钮点击没反应，调试半天才发现。

> 💡 用第三方组件库的代价是要适应它的约定。每个库的「坑」都不一样，但都会有。


### 8.2 axios 的统一调用

整个前端没有封装 axios 实例，每个组件直接：

```js
await axios.get('http://127.0.0.1:8000/api/recommend/?user_id=' + uid)
await axios.post('http://127.0.0.1:8000/api/users/login/', form)
```

这种「直接调用、域名硬编码」的写法在生产项目里是**反模式**，应该封装一个 `request.js` 工具，统一处理 baseURL、请求拦截器、响应拦截器、错误处理。

为什么没封装：毕设阶段后端域名只有一个（`localhost:8000`），改一次相当于全局替换；接口数量也不多（22 个），手动维护成本可接受。

**生产环境改造方向**：
- 抽 `utils/request.js`
- 把 baseURL 写到 `.env.development` / `.env.production`
- 请求拦截器统一加 Authorization Header
- 响应拦截器统一处理 401/500


### 8.3 状态管理：直接用 localStorage 没用 Pinia

整个项目没引入 Pinia（也没引入 Vuex）。用户信息（user_id、role 等）直接读 localStorage：

```js
const user_id = localStorage.getItem('user_id')
```

为什么没用 Pinia：

- 项目里需要跨组件共享的状态只有「当前登录用户信息」这一项
- 这一项用 localStorage 就够了：天然持久化、不需要额外配置
- 引入 Pinia 会增加心智成本：每个组件都要 `useUserStore()` 后再访问

**什么时候必须用 Pinia**：当你有以下场景之一时再引入：
- 多个组件共享一个会变化的状态（如购物车）
- 需要派生计算属性（如购物车合计金额）
- 状态变化要触发多个组件同步更新

毕设场景下都不满足。


### 8.4 长得很像的代码：列表页的模板

求职者端的 `JobList.vue`、`RecommendJobs.vue`、`MyApplications.vue` 三个页面的整体结构几乎一样：

```
顶部筛选条 → el-table 列表 → 操作按钮列 → 点击行进入详情
```

我没做组件抽象。三个页面长得像但**业务逻辑差异不小**：JobList 要支持搜索过滤、RecommendJobs 要显示匹配度、MyApplications 要显示投递状态。强行抽象成一个 `JobTable.vue` 会变成参数巨多的「**God Component**」。

最终决定：**保持冗余，但保持一致**。三个页面的结构相同、命名风格相同、样式类相同，让看代码的人能快速从一个页面切换到另一个。

> 💡 提前抽象（premature abstraction）比重复代码更危险——重复代码看着难看但你能改，过度抽象让你想改改不动。

## 第九章 关键技术决策回顾

回头看整个项目，有几个关键决策值得单独回顾——既包括做对的，也包括差点做错被自己刹车的。


### 9.1 决策一：Django 而不是 FastAPI

**选择**：Django + DRF

**理由**：项目是典型的多角色、多表关联、CRUD 密集的 Web 系统，Django 的 ORM + Admin + Migration 全套组合拳完美匹配。

**回头看**：选对了。如果用 FastAPI，光是 ORM（要单独引入 SQLAlchemy）和 Migration（要单独引入 Alembic）就要花掉至少一周时间配置。Django 一键启动，开发效率提升明显。


### 9.2 决策二：SQLite 而不是 MySQL

**选择**：SQLite

**理由**：数据规模匹配 + 零配置 + Django 透明切换。

**回头看**：选对了。整个项目 clone 下来 `python manage.py runserver` 就能跑，答辩老师本地复现零成本。如果用 MySQL，老师还得先装 MySQL Server、建库、改配置文件，体验差很多。


### 9.3 决策三：TF-IDF 而不是 BERT

**选择**：TF-IDF + 余弦相似度

**理由**：毕设场景下数据量小、可解释性强、不需要训练。

**回头看**：选对了。BERT 的语义理解能力确实更强，但：
- 模型文件几百兆，毕设答辩演示时部署麻烦
- 推理慢，需要 GPU
- 可解释性弱——「为什么这个岗位匹配度高」答不上来
- 老师如果问「你训练了多久」，TF-IDF 直接说「不需要训练」就完了

毕设的关键不是用了多先进的算法，而是**讲清楚为什么用这个算法**。


### 9.4 决策四：把简历和岗位合并到一个 corpus

**选择**：`corpus = [resume_text] + job_texts`，统一 fit_transform

**理由**：保证简历向量和岗位向量在同一特征空间。

**回头看**：必须的。这是 TF-IDF 工程实现里最容易踩的坑，分开 fit 就废了。


### 9.5 决策五：list() 强制实例化 QuerySet

**选择**：`jobs = list(Job.objects.filter(status='approved'))`

**理由**：避免 Django ORM 懒加载导致的重复查库。

**回头看**：写库代码的好习惯。毕设场景下性能差异不明显，但养成习惯很重要。


### 9.6 决策六：候选人匹配度函数不复用推荐主函数

**选择**：`calc_resume_job_score` 和 `get_recommendations` 保持两份独立函数

**理由**：两份函数虽然算法一样，但语义完全不同——一个返回列表一个返回单值，一个用于求职者端一个用于 HR 端。强行合并需要传一堆控制参数，反而更难维护。

**回头看**：选对了。「Don't Repeat Yourself」是好原则，但前提是「**重复的是相同业务逻辑**」。两个独立业务场景的相似代码，独立才是最优解。


### 9.7 决策七：差点做错——审核工作流

**差点做的事**：原本想做完整的「审核工作流」——HR 提交岗位后可以撤回、管理员可以反复审核、被拒后允许 HR 修改重新提交。还想加邮件通知。

**为什么砍掉**：

- 审核工作流是企业 SaaS 系统的标配，但毕设场景下答辩只看主流程
- 邮件通知需要配 SMTP，调试时间长，且答辩时无法演示（没邮箱接收）
- AI 助手在我开始失控时提醒过：「**毕设的常见陷阱是无限加功能、永远没有定稿的那天**」

**砍掉后保留的版本**：HR 提交岗位 → 默认 pending → 管理员一键 approve/reject → 状态变更。一根直线，没有分支。

**回头看**：砍得对。砍掉后能腾出时间打磨推荐算法（毕设的真正特色点），加上文档撰写，效果远比「功能多但每个都浅尝辄止」要好。

> 💡 做毕设最难的不是「加功能」，是「**决定不加功能**」。前者拼体力，后者拼判断力。


### 9.8 决策八：差点做错——前端 UI 框架

**差点做的事**：一开始打算用 Tailwind CSS 自己手写所有组件，理由是「这样显得我前端功底深」。

**为什么砍掉**：试着写了一个 Login 页就放弃了。Tailwind 适合**有设计稿**的项目——美工给你像素级稿子，你 1:1 还原。我没有美工，自己设计的样式很快就「越改越丑」。

**最终方案**：Element Plus 一把梭，UI 一致性瞬间提升。

**回头看**：选对了。前端不是这个毕设的考核点，把时间花在「自己造 UI」上是浪费。

> 💡 区分「**核心战场**」和「**辅助战场**」。在核心战场（推荐算法、数据模型）多投入，在辅助战场（前端 UI、错误页面）用现成方案。

## 第十章 求职招聘业务知识补充

这一章是写给「**虽然写了一个招聘系统、但对真实招聘业务还不太懂的自己**」看的。把项目里涉及到的业务知识系统化整理一遍。


### 10.1 招聘系统的核心业务环节

完整的招聘业务通常包含以下环节：

| 环节 | 主要参与方 | 主要交付物 |
|---|---|---|
| 用人需求确定 | 用人部门 + HR | 岗位 JD（Job Description） |
| 岗位发布 | HR | 岗位上线（公司官网、招聘平台） |
| 简历收集 | HR | 简历池 |
| 简历筛选 | HR | 通过筛选的候选人名单 |
| 面试 | HR + 用人部门 | 面试评估表 |
| Offer 沟通 | HR | 入职 Offer |
| 入职 | HR + 行政 | 入职手续办理 |

本项目主要覆盖**岗位发布、简历收集、简历筛选**三个环节，**面试、Offer、入职**通过候选人状态机简化抽象（hired 状态 ≈ 完成入职）。


### 10.2 岗位 JD 的标准字段

参考主流招聘平台（智联、BOSS 直聘、前程无忧），一个标准的岗位 JD 通常包含以下字段：

**基础信息**：岗位名称、所属企业、工作城市、薪资范围、招聘人数

**任职要求**：学历要求、工作经验要求、专业要求、年龄要求（部分场景）

**岗位描述**：岗位职责、任职要求、加分项

**福利**：五险一金、年终奖、补充医疗、节日福利、培训机会

**联系方式**：投递邮箱 / 在线投递入口

本项目的 `Job` 表覆盖了前三类，福利与联系方式简化处理（投递通过 `apply_job` 接口完成，不需要邮箱）。


### 10.3 简历的标准结构

参考 LinkedIn、智联简历、超级简历等主流简历模板，一份标准简历的结构：

**基础信息**：姓名、电话、邮箱、求职意向

**教育经历**：学校、专业、学历、入学/毕业时间、绩点（可选）、主修课程（可选）

**工作经历 / 实习经历**：公司、岗位、起止时间、工作内容、业绩

**项目经历**：项目名称、项目角色、技术栈、项目描述、项目成果

**专业技能**：编程语言、框架、工具、外语水平

**证书与荣誉**：英语证书、计算机证书、行业证书、获奖记录

**自我评价**：性格特点、职业目标

本项目的 `Resume` 表覆盖了全部 7 类，与主流简历模板字段基本一一对应。这也是为什么 `Resume` 表有 19 个业务字段的原因——少一个都不算「完整简历」。


### 10.4 中文简历与英文简历的差异

**中文简历**：

- 通常 1-2 页 A4 纸
- 包含照片（部分企业要求）、年龄、籍贯等个人信息
- 教育经历完整列出
- 强调「具体做了什么」

**英文简历（CV / Resume）**：

- 通常 1 页 A4 纸
- 不放照片（避免就业歧视）、不写年龄/籍贯
- 教育经历只列最近几段
- 强调「具体的量化成果」（如 "Improved API response time by 40%"）

本项目按中文简历习惯设计字段（含照片预留位、含期望薪资字段），未来如果做国际化需要按英文简历模板调整。


### 10.5 推荐算法在招聘行业的实际应用

主流招聘平台都用了推荐算法，但每家侧重点不同：

| 平台 | 推荐特色 |
|---|---|
| BOSS 直聘 | 强调实时性，求职者与 HR 双向互推 |
| 智联招聘 | 多维度筛选（行业、职能、薪资），推荐与搜索结合 |
| 前程无忧 | 简历投递历史 + 简历内容混合推荐 |
| LinkedIn | 基于职业网络的社交推荐（"People You May Know" + "Jobs You May Like"） |

我的毕设属于「**单纯基于简历内容的推荐**」，对标的是 BOSS 直聘和智联招聘**早期**的推荐策略——这类策略简单、可解释、对冷启动用户友好，是任何推荐系统的起点。后期所有大平台都引入了协同过滤、行为序列建模、深度学习等更复杂的模型，但基础的 TF-IDF 内容匹配几乎都还作为「保底通道」存在。

## 第十一章 AI 协作的反思

这个项目我延续了上一个项目（HCM 实施助手）养成的 AI 协作习惯：大部分代码由 AI 助手生成，但所有方向决策由我把握。做完毕设后，我对「人和 AI 协作」又有一些新的体会。


### 11.1 这次和上次的差异

**上次（HCM 助手）**：项目是从零开始的探索，AI 帮我快速验证想法、写模板代码。

**这次（毕设）**：项目有明确的论文要求和指导老师反馈，AI 的角色更像「**结构化辅助**」——帮我把分散的需求整理成可执行的方案，把指导老师的批注转化为具体的修改点。

这次最有价值的几次 AI 协作：

- **论文修订**：指导老师返回的批注分散在 PDF 各处，我把所有批注整理给 AI，让它生成结构化的修改方案。AI 甚至会主动问「你想要的章节排序是哪种」，确保不漏掉一个改动。
- **算法推导**：我让 AI 用真实数值演算一遍 TF-IDF + 余弦相似度的完整流程，验证我论文里写的样例数据是否准确（结果发现论文表 4-13 的数值和 sklearn 实际计算结果不一致——这个发现救了我一命）。
- **文档生成**：本文档、推荐算法说明文档、答辩 PPT 大纲都是由 AI 协作生成的。


### 11.2 AI 在这次项目里的局限

**业务理解的边界**：AI 不知道郑州轻工业大学毕设的具体格式要求（行距、字号、页眉、参考文献格式），我需要把这些规范明确告诉它。每个学校的标准都不同，AI 没有这个先验。

**学术规范的边界**：AI 写出来的论文段落容易过于「**完整**」——什么都讲一点、面面俱到。学术写作恰恰相反——选定一个角度深入挖。我需要主动指挥它「**这一段只讲冷启动，不要扯到其他**」。

**自我验证的边界**：AI 给出的代码示例、数值演算、引用文献，**必须自己跑一遍 / 算一遍 / 查一遍**。前面提到的「论文表 4-13 数值错误」就是个反例——AI 当时给了一个看起来合理的数值，但实际上是它估算的，不是真算的。直到我后来自己用 sklearn 跑了一遍才发现。


### 11.3 一个具体的案例

毕设答辩临近时，我一度想给系统再加一个「**简历得分预测**」功能——根据简历内容预测一个总分，让求职者了解简历完整度。我把这个想法告诉 AI，它没有立刻答应，反而问了几个问题：

> 💡 这个功能和你毕设的核心主题「内容推荐」是什么关系？它会不会让你的论文焦点从「推荐算法」分散到「评分模型」？答辩老师如果问「评分模型用了什么算法」，你准备怎么答？

这一连串反问让我冷静下来。最终决定不做这个功能——它和主题关联不强，又会增加新的算法负担。

> 💡 好的 AI 助手不是「无条件答应」的工具，而是「会反问你」的协作者。会反问的 AI 比啥都顺着你说的 AI 有用得多。


### 11.4 与 AI 协作的几个原则（延续上一项目）

经过两个项目，我提炼出几条比较稳定的原则：

> 💡 **原则一：先定边界，再说目标**。「这个项目要做什么」比「这个项目能做什么」重要 10 倍。

> 💡 **原则二：每个代码块自己跑一遍**。不要假设「AI 说能跑就能跑」。

> 💡 **原则三：数值必须自己验算**。AI 给的数值可能是它「估算」的，不是真算的。

> 💡 **原则四：把指导老师的反馈完整提供给 AI**。零散的反馈会得到零散的方案，结构化的反馈才能得到结构化的修订。

> 💡 **原则五：拒绝功能蔓延**。AI 容易「贪多」，要主动喊停。

## 第十二章 后续规划与反思


### 12.1 如果再多一个月

如果毕设答辩之后还有时间继续打磨，我会按以下顺序加东西：

**优先级 1：把推荐算法升级为「TF-IDF + 行为加权」**

当前 TF-IDF 完全没用上行为数据。可以做一个简单的加权：求职者过去 30 天投递过的岗位类型，作为权重叠加到推荐分数上。不需要训练模型，只需要简单的统计。

**优先级 2：把每次重算改为「向量缓存」**

把所有 approved 岗位的 TF-IDF 向量预先计算好缓存到内存或 Redis。岗位新增/修改时增量更新缓存。性能能提升一个数量级。

**优先级 3：补充审计日志**

把 `audit_log` 表真正用起来——每次岗位审核、用户封禁都写一条日志。便于答辩时演示「**这个系统是有完整审计能力的**」。

**优先级 4：补单元测试**

当前一个测试都没写（`tests.py` 是空的）。要把推荐算法的核心场景（冷启动、空简历、空岗位池）和身份认证流程的关键路径都测起来。


### 12.2 如果重头再来

**会重新做的事**：

- **从一开始就用 JWT**。后期再升级要改前后端大量代码，前期就用反而轻松。
- **从一开始就分环境配置**。`settings.py` 拆成 `base.py` + `dev.py` + `prod.py`。
- **从一开始就写文档**。这份讲解文档要是边做边写，每个章节都更鲜活——现在回忆某些决策细节已经有点模糊了。

**不会改变的事**：

- **算法选 TF-IDF**。再做一遍我还是选它，理由不变。
- **数据库选 SQLite**。毕设场景下没有比这更好的。
- **前端 UI 用 Element Plus**。不要在辅助战场浪费时间。


### 12.3 这个项目教会我什么

代码层面：

- Django ORM 的懒加载机制和 `select_related` 优化
- scikit-learn 的 TF-IDF 向量化使用细节
- Vue Router 嵌套路由设计
- Element Plus 的常见坑

思路层面：

- **简化是一种能力**。知道什么不做比知道什么做更难。
- **可解释性比先进性更重要**。能讲清楚的算法比讲不清楚的算法更适合毕设。
- **写文档不是负担，是「**面向未来的礼物**」**。三个月后回来改代码时，你会感谢现在的自己。

业务层面：

- 招聘行业的全流程业务环节
- 简历与岗位描述的标准字段结构
- 主流招聘平台的推荐策略差异
